# インデックス有無によるクエリ実行計画の比較

## 📋 背景

PostgreSQL のインデックスが実際にどれほどクエリ速度に影響するかを体感するため、
`users` テーブルに 10,000 件のデータを投入し、インデックスの有無で `EXPLAIN ANALYZE` の結果を比較した。

### なぜ `name` カラムで試したか

最初は `email` カラムで検証しようとしたが、`email` には `UNIQUE` 制約が設定されており、
PostgreSQL は UNIQUE 制約に対して**自動的にインデックスを作成する**仕様になっている。
そのため `email` カラムでは「インデックスなし」の状態が作れず、最初から `Index Scan` になっていた。

UNIQUE 制約のない `name` カラムに切り替えることで、`Seq Scan` → `Index Scan` の変化を正しく確認できた。

> 💡 **学び：** `\di テーブル名*` でインデックス一覧を確認する習慣をつけることで、
> 暗黙のインデックス（UNIQUE制約由来）の存在に気づける。

---

## 🔬 実施内容

### 環境

| 項目       | 内容                     |
| ---------- | ------------------------ |
| PostgreSQL | 16                       |
| テーブル   | `users`（10,003件）      |
| 検索カラム | `name`（UNIQUE制約なし） |
| 検索値     | `'田中5000'`             |

### 手順

```sql
-- 1. 10,000件のデータを生成
INSERT INTO users (name, email)
SELECT '田中' || i, 'user' || i || '@example.com'
FROM generate_series(1, 10000) AS i
ON CONFLICT DO NOTHING;

-- 2. インデックスなしで実行計画を確認
EXPLAIN ANALYZE
SELECT * FROM users WHERE name = '田中5000';

-- 3. インデックスを作成
CREATE INDEX idx_users_name ON users(name);

-- 4. インデックスありで実行計画を再確認
EXPLAIN ANALYZE
SELECT * FROM users WHERE name = '田中5000';
```

## 📊 結果

### インデックスなし（Seq Scan）

```
Seq Scan on users  (cost=0.00..219.04 rows=1 width=42)
                   (actual time=0.582..1.090 rows=1 loops=1)
  Filter: ((name)::text = '田中5000'::text)
  Rows Removed by Filter: 10002
Planning Time: 0.285 ms
Execution Time: 1.118 ms
```

**読み方：**
- `Seq Scan`：テーブルの全件を先頭から順番にスキャンしている
- `Rows Removed by Filter: 10002`：10,002件を読み込んで1件だけ残した（無駄が多い）
- 実行時間：**1.118 ms**

### インデックスあり（Index Scan）

```
Index Scan using idx_users_name on users  (cost=0.29..8.30 rows=1 width=42)
                                          (actual time=0.016..0.016 rows=1 loops=1)
  Index Cond: ((name)::text = '田中5000'::text)
Planning Time: 0.085 ms
Execution Time: 0.024 ms
```

**読み方：**
- `Index Scan`：B-Tree インデックスを使って目的のレコードに直接アクセスしている
- `Rows Removed by Filter` の記載なし：無駄なスキャンがなかった
- 実行時間：**0.024 ms**

### 比較まとめ

| 項目         | インデックスなし | インデックスあり   | 改善率               |
| ------------ | ---------------- | ------------------ | -------------------- |
| スキャン方式 | Seq Scan（全件） | Index Scan（直接） | —                    |
| スキャン件数 | 10,003件         | 1件                | 約 **10,000倍** 削減 |
| 実行時間     | 1.118 ms         | 0.024 ms           | 約 **46倍** 高速化   |

## 🔍 詰まったこと・解決方法

### UNIQUE制約で最初からIndex Scanになっていた問題

**状況：** `email` カラムで試したところ、インデックスを作成する前から `Index Scan` になっていた。

**原因：** `email VARCHAR(255) UNIQUE NOT NULL` の `UNIQUE` 制約により、
PostgreSQL がテーブル作成時点で自動的にインデックス（`users_email_key`）を作成していた。

**解決方法：** `\di users*` でインデックス一覧を確認し、UNIQUE制約由来のインデックスが存在することを発見。
UNIQUE制約のない `name` カラムに切り替えて検証した。

```sql
-- インデックス一覧の確認コマンド
\di users*

-- 結果（2つのインデックスが存在していた）
-- users_email_key  → UNIQUE制約が自動作成したもの
-- users_pkey       → PRIMARY KEY が自動作成したもの
-- idx_users_name   → 今回手動で作成したもの
```

## 💡 気づき

1. **インデックスの仕組み**
   PostgreSQL のデフォルトインデックスは B-Tree 構造。
   検索計算量が `O(N)`（全件スキャン）から `O(log N)`（二分探索）に改善される。

2. **UNIQUE制約と暗黙のインデックス**
   UNIQUE制約・PRIMARY KEY には PostgreSQL が自動でインデックスを作成する。
   重複してインデックスを作ると書き込み時のオーバーヘッドが増えるため `\di` で確認する。

3. **インデックスのトレードオフ**
   インデックスは SELECT を高速化する一方、INSERT・UPDATE・DELETE のたびに
   インデックスの更新も発生するため書き込みが遅くなる。
   「読み取り頻度が高いカラム」「WHERE句やJOINで頻繁に使うカラム」に絞って貼るのが原則。

4. **EXPLAIN ANALYZE の使い方**
   - `EXPLAIN`：実行計画のみ（実際には実行しない）
   - `EXPLAIN ANALYZE`：実際に実行して実測値も表示
   - `Seq Scan` が出たらインデックス不足を疑う
   - `cost` は推定値、`actual time` が実測値
