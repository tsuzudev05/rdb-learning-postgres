# 分離レベル（READ COMMITTED）動作確認結果

## 📋 背景

PostgreSQL のトランザクション分離レベルが実際にどのように動作するかを体感するため、
2つのセッションを同時に実行し、`READ COMMITTED` における**ノンリピータブルリード**を確認した。

ACID 特性の「Isolation（分離性）」は分離レベルによって挙動が異なる。
今回は最も基本的な `READ COMMITTED` を検証し、`REPEATABLE READ` との違いを理解することを目的とした。

### 分離レベルの種類と特性

| 分離レベル | ダーティリード | ノンリピータブルリード | ファントムリード |
|---|---|---|---|
| READ UNCOMMITTED | 発生する | 発生する | 発生する |
| **READ COMMITTED** | **発生しない** | **発生する** ← 今回確認 | **発生する** |
| REPEATABLE READ | 発生しない | 発生しない | 発生する※ |
| SERIALIZABLE | 発生しない | 発生しない | 発生しない |

※PostgreSQL の REPEATABLE READ はファントムリードも発生しない

## 🔬 実施内容

### 環境

| 項目 | 内容 |
|---|---|
| PostgreSQL | 16 |
| テーブル | `tasks`（4件） |
| 対象レコード | id=2（JOINを理解する） |
| 分離レベル | READ COMMITTED |
| 確認内容 | ノンリピータブルリードの発生 |

### 検証シナリオ

```
セッションA: BEGIN → 1回目SELECT → （待機） → 2回目SELECT → COMMIT
セッションB:                    BEGIN → UPDATE → COMMIT
```

### 手順

```sql
-- セッションA（1回目SELECT）
BEGIN;
SET TRANSACTION ISOLATION LEVEL READ COMMITTED;
SELECT id, title, status FROM tasks WHERE id = 2;
-- ここでセッションBの完了を待機

-- セッションB（AのSELECT後にUPDATE・COMMIT）
BEGIN;
UPDATE tasks SET status = 'done' WHERE id = 2;
COMMIT;

-- セッションA（2回目SELECT・BのCOMMIT後）
SELECT id, title, status FROM tasks WHERE id = 2;
COMMIT;
```

### 実装方法

2つのターミナルを手動で操作する代わりに、**名前付きパイプ（FIFO）** を使って
2つのセッションを同期させるシェルスクリプト（`benchmark_isolation.sh`）を作成した。

```
セッションA（バックグラウンド）          セッションB（メイン）
─────────────────────────────────────────────────────────
BEGIN・1回目SELECT実行
「Bを待機中...」         ←── 同期 ──→   BEGIN・UPDATE・COMMIT
「B完了を検知」          ←── 通知 ───   「Aに通知」
2回目SELECT・COMMIT
```

## 📊 結果

### セッションAの1回目SELECT（BがCOMMITする前）

```
-[ RECORD 1 ]----------
id     | 2
title  | JOINを理解する
status | in_progress    ← Bがまだ変更していない状態
```

### セッションBのUPDATE・COMMIT

```
BEGIN
UPDATE 1                ← 1件更新
COMMIT                  ← in_progress → done に永続化
```

### セッションAの2回目SELECT（BがCOMMITした後）

```
-[ RECORD 1 ]----------
id     | 2
title  | JOINを理解する
status | done           ← BのCOMMITした値が見えている ✅
```

### 比較まとめ

| タイミング | セッションAが見たstatus | 変化 |
|---|---|---|
| BがCOMMITする前（1回目SELECT） | in_progress | — |
| BがCOMMITした後（2回目SELECT） | done | 変わった ✅ |

**同一トランザクション内で同じSELECTを2回実行したのに結果が変わった**
→ これが **ノンリピータブルリード** である

---

## 🔍 詰まったこと・解決方法

### WARNING: there is no transaction in progress

**状況：** 2回目のSELECT後に `COMMIT` を実行したとき以下の警告が出た。

```
WARNING: there is no transaction in progress
```

**原因：** シェルスクリプトでセッションAのSQLを複数のpsql呼び出しに分割したため、
2回目の `SELECT` が別セッションとして実行されトランザクションが存在しない状態になった。

**動作への影響：** なし。ノンリピータブルリードの確認結果は正しく取得できている。

**解決方針：** 2回目のSELECTとCOMMITを同一psqlセッション内で実行するよう修正予定。

## 💡 気づき

1. **ノンリピータブルリードとは**
   同一トランザクション内で同じ行を2回SELECTしたとき、
   別トランザクションのCOMMITによって結果が変わる現象。
   `READ COMMITTED` では発生する。

2. **REPEATABLE READとの違い**
   `REPEATABLE READ` にすると、トランザクション開始時点のスナップショットを
   読み続けるため、Bが `COMMIT` しても2回目のSELECTは `in_progress` のままになる。
   データの一貫性が求められる場面（在庫確認・残高確認など）では `REPEATABLE READ` が適切。

3. **DDD × 分離レベルの関係**
   Repository パターンで複数の集約を一括取得する際、
   途中で別トランザクションが値を変更すると整合性が崩れる可能性がある。
   ユースケースの性質に応じて分離レベルを使い分けることが重要。

4. **名前付きパイプ（FIFO）による同時実行の再現**
   `mkfifo` でパイプを作成しセッション間の同期を実現。
   実際のデータベースの競合状態をシェルスクリプトで再現する手法として有効。
