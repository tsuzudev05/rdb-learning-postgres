# デッドロック動作確認結果

## 📋 背景

PostgreSQL のデッドロックが実際にどのように発生し、どのように解消されるかを体感するため、
2つのセッションを真の同時実行で動かし、**デッドロックの再現と回避策**を確認した。

分離レベルの確認では「別セッションの書き込みが見えるか」を観察したが、
デッドロックは**2つのセッションが互いに相手のロックを待ち合う**ことで発生する別の問題である。
アプリ側でのリトライ設計や、そもそもデッドロックを起こさない実装パターンを理解することを目的とした。

---

## 🔬 実施内容

### 環境

| 項目       | 内容                                         |
| ---------- | -------------------------------------------- |
| PostgreSQL | 16                                           |
| テーブル   | `tasks`（id=1, id=2）                        |
| 実行方法   | Python `threading` で2スレッドを真の同時実行 |
| 確認内容   | デッドロック発生・自動検知・回避策           |

### 検証シナリオ

```
Part 1: デッドロックを意図的に発生させる
  セッションA: id=1 をロック → id=2 をロック待ち
  セッションB: id=2 をロック → id=1 をロック待ち
  → 互いに相手のロックを待ち合うのでデッドロック

Part 2: 回避策（ロック取得順を統一する）
  セッションA: id=1 → id=2 の順でUPDATE（昇順）
  セッションB: id=1 → id=2 の順でUPDATE（昇順・同じ順番）
  → 同じ順番なので競合は起きるが待ち合いにはならない
```

### 実装方法

名前付きパイプではなく **Python の `threading.Event`** を使って2スレッドを同期させた。
`Event.set()` / `Event.wait()` でセッション間のタイミングを制御し、
確実にデッドロックが起きる状態を作り出している。

```python
a_locked = threading.Event()  # AがROW1をロックしたらセット
b_locked = threading.Event()  # BがROW2をロックしたらセット

# セッションA
cur.execute("UPDATE tasks SET status = 'in_progress' WHERE id = 1")
a_locked.set()    # AがROW1をロック完了 → Bに通知
b_locked.wait()   # BがROW2をロックするまで待機
cur.execute("UPDATE tasks SET status = 'in_progress' WHERE id = 2")  # ← ここでデッドロック

# セッションB
a_locked.wait()   # AがROW1をロックするまで待機
cur.execute("UPDATE tasks SET status = 'done' WHERE id = 2")
b_locked.set()    # BがROW2をロック完了 → Aに通知
cur.execute("UPDATE tasks SET status = 'done' WHERE id = 1")          # ← ここでデッドロック
```

## 📊 結果

### 事前確認：対象データ

```
-[ RECORD 1 ]--------------
id     | 1
title  | インデックスを学ぶ
status | todo
-[ RECORD 2 ]--------------
id     | 2
title  | JOINを理解する
status | todo
```

---

### Part 1：デッドロックを意図的に発生させる

**実行ログ**

```
[セッションA] BEGIN → id=1 をUPDATE（ロック取得）
[セッションA] id=1 ロック完了 → セッションBを待機中...
[セッションB] BEGIN → id=2 をUPDATE（ロック取得）
[セッションB] id=2 ロック完了 → セッションAに通知
[セッションA] id=2 のUPDATEを要求（Bと競合 → デッドロック待ち）
[セッションB] id=1 のUPDATEを要求（Aと競合 → デッドロック待ち）
[セッションA] ERROR: deadlock detected → 自動ROLLBACK   ← Aが犠牲者に選ばれた
[セッションB] COMMIT完了
```

**Part 1 後のデータ**

```
id=1 | インデックスを学ぶ | status=done   ← Bの変更のみ反映
id=2 | JOINを理解する     | status=done   ← Bの変更のみ反映
```

**読み方：**
- AとBが互いに相手のロックを待ち合い、PostgreSQLが約1秒でデッドロックを自動検知
- AとBのどちらかを**犠牲者（victim）**として選び強制 ROLLBACK する
- 今回はAが犠牲者に選ばれ、BのみがCOMMITされた
- どちらが犠牲者になるかはPostgreSQLが内部で決定し、アプリ側では制御できない

---

### Part 2：回避策（ロック取得順を統一する）

**実行ログ**

```
[セッションA] BEGIN → id=1 → id=2 の順でUPDATE（昇順）
[セッションA] COMMIT完了
[セッションB] BEGIN → id=1 → id=2 の順でUPDATE（昇順・同じ順番）
[セッションB] COMMIT完了（デッドロックなし）
```

**Part 2 後のデータ**

```
id=1 | インデックスを学ぶ | status=done   ← 両方が正常に反映
id=2 | JOINを理解する     | status=done   ← 両方が正常に反映
```

**読み方：**
- 両セッションが id=1 → id=2 の同じ順番でロックを取得するため、待ち合いが発生しない
- AがCOMMITするまでBはid=1のロック待ちになるが、それはデッドロックではなく**通常の待機**

---

### 比較まとめ

| シナリオ | ロック取得順                 | 結果                            |
| -------- | ---------------------------- | ------------------------------- |
| Part 1   | A: id=1→2、B: id=2→1（逆順） | デッドロック発生 → 片方ROLLBACK |
| Part 2   | A: id=1→2、B: id=1→2（同順） | 正常完了 ✅                      |

## 🔍 詰まったこと・解決方法

### `psycopg2` が未インストールでスクリプトが動かなかった

**状況：** `python3 << 'PYEOF' ... PYEOF` のヒアドキュメント部分でスクリプトが落ちた。

```
ModuleNotFoundError: No module named 'psycopg2'
```

**原因：** コンテナ環境に `psycopg2` がインストールされていなかった。

**解決までの試行錯誤：**

```bash
# pip3 コマンド自体がない
$ pip3 install psycopg2-binary
bash: pip3: command not found

# python3 の pip モジュールもない
$ python3 -m pip install psycopg2-binary
No module named pip

# apt キャッシュが古くてパッケージが見つからない
$ apt-get install -y python3-psycopg2
E: Unable to locate package python3-psycopg2

# apt-get update でキャッシュを更新してから再試行 → 解決
$ apt-get update && apt-get install -y python3-psycopg2
```

**解決方法：** `apt-get update` でパッケージリストを更新してから `python3-psycopg2` をインストール。
`pip` が使えない環境では OS のパッケージマネージャーで Python ライブラリを入れる方法が有効。

## 💡 気づき

1. **デッドロックの本質**
   2つのトランザクションが「相手が持っているロックを待ち合う」状態になること。
   AがBを待ち、BがAを待つ → 永遠に進まないため、外部からの介入（強制ROLLBACK）が必要になる。

2. **PostgreSQL の自動検知**
   PostgreSQL は約1秒間隔でデッドロックを検知し、`deadlock_detected` エラーを発生させる。
   犠牲者に選ばれたセッションは自動で ROLLBACK され、もう片方が処理を続行できる。
   アプリ側は `deadlock_detected` を受け取ったらトランザクション全体をリトライする実装が必要。

3. **回避策の本質：ロック取得順の統一**
   複数行を更新する場合は**常に同じ順番**（例：id 昇順）でロックを取得すれば、
   待ち合いは起きても「一方が先に進める」ため循環しない。

4. **`threading.Event` による同時実行の再現**
   `Event.set()` / `Event.wait()` を使うと、スレッド間で「ここまで終わったら通知」という
   同期ポイントを作れる。今回のように「デッドロックが発生する瞬間」を制御するのに有効。

5. **DDD × デッドロックの関係**
   Repository パターンで複数の集約をまたいで更新する場合、
   取得・更新の順番が一定でないとデッドロックのリスクがある。
   ユースケース層でどの順番でRepositoryを呼ぶかを統一しておくことが設計上重要になる。
